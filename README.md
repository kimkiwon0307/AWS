## ☁️ 1.1 클라우드 컴퓨팅 (Cloud Computing)

### 1.1.1 클라우드 컴퓨팅이란?
* **정의 :** 자체적인 물리 서버를 구축하는 대신, 인터넷망을 통해 IT 자원(서버, 스토리지, 네트워크 등)을 **필요한 만큼 빌려 쓰고 비용을 지불하는 서비스**입니다.
* **비유하자면 :** * **온프레미스(On-Premise) :** 대지 매입부터 벽돌 쌓기, 전기/수도 공사까지 직접 수행하여 **집을 짓는 것**
  * **클라우드(Cloud) :** 이미 완벽하게 인프라가 갖춰진 공간에 체크인하여 편하게 머무는 **호텔을 이용하는 것**

### ⚖️ 온프레미스 vs 클라우드 한눈에 비교
| 비교 항목 | 온프레미스 (전통적 방식) | 클라우드 컴퓨팅 (현대적 방식) |
| :--- | :--- | :--- |
| **초기 비용** | 서버 장비 구매, 데이터 센터 구축으로 **막대한 초기 비용** 발생 | 장비 구매가 없으므로 **초기 비용 0원에 수렴** |
| **구축 속도** | 장비 발주, 배송, 서버실 실장까지 **수주~수개월** 소요 | 콘솔 클릭 몇 번으로 **수분 이내**에 인프라 생성 |
| **확장성** | 트래픽 증가 시 추가 서버를 사서 꽂아야 하므로 **증설이 매우 느림** | 클릭 또는 설정에 따라 서버가 실시간 자동 증가 (**Auto Scaling**) |
| **유지 관리** | 전기세, 공조 장치, 고장 난 부품 교체 등 **직접 관리 부담** | 물리 하드웨어 관리 및 인프라 보안은 **클라우드 공급사(AWS)가 전담** |

### 🚀 1.1.2 클라우드 컴퓨팅의 핵심 이점
* **자본 비용의 가변 비용화 :** 대규모 인프라를 미리 구매할 필요가 없어 비즈니스 리스크를 최소화합니다.
* **엄청난 규모의 경제 :** 수많은 가입자가 자원을 공유하므로, 개인이 구축하는 것보다 훨씬 저렴한 단가로 고성능 인프라를 이용할 수 있습니다.
* **용량 예측 불필요 :** 트래픽 수요를 미리 예측해 과도하게 장비를 사둘 필요 없이, 실제 유입량에 맞추어 유연하게 대응합니다.
* **글로벌 배포 가속화 :** 몇 번의 클릭만으로 전 세계(미국, 유럽, 아시아 등)에 서버를 즉시 배포하여 글로벌 서비스를 제공할 수 있습니다.

### 📦 1.1.3 클라우드 컴퓨팅 서비스 유형 (IaaS / PaaS / SaaS)
공급사(AWS)가 어디까지 관리해 주고, 엔지니어가 어디서부터 책임지느냐에 따른 분류입니다.

* **IaaS (Infrastructure as a Service, 인프라 제공 서비스)**
  * **설명 :** 가상화된 하드웨어 자원(서버, 네트워크, 스토리지)만 빌려주는 형태입니다. OS 선택부터 웹 서버, 프로그램 설치 및 환경 설정까지 **엔지니어가 가장 자유롭게 제어**할 수 있습니다.
  * **AWS 대표 서비스 :** **Amazon EC2**, Amazon VPC
  * **실무 예시 :** EC2 서버를 생성한 뒤 Ubuntu OS 위에 Nginx와 Docker를 직접 커맨드로 설치하고 아키텍처를 빌드함.
* **PaaS (Platform as a Service, 플랫폼 제공 서비스)**
  * **설명 :** 밑바탕이 되는 인프라와 OS, 런타임 환경까지 인프라 가상화 플랫폼이 전부 알아서 관리해 줍니다. 개발자는 하드웨어 신경 세팅 없이 **오직 '애플리케이션 코드'만 업로드**하면 됩니다.
  * **AWS 대표 서비스 :** **AWS Elastic Beanstalk**, AWS Lambda
  * **실무 예시 :** 개발한 백엔드 소스 코드를 압축해서 업로드하면, AWS가 알아서 서버 대수 조절과 로드 밸런싱을 수행함.
* **SaaS (Software as a Service, 소프트웨어 제공 서비스)**
  * **설명 :** 클라우드 공급업체가 인프라부터 최종 소프트웨어 애플리케이션까지 **100% 완성하여 서비스 형태로 제공**하는 모델입니다. 사용자는 별도의 설치 없이 계정 로그인만 해서 기능을 이용합니다.
  * **대표 예시 :** Gmail, Slack, Notion, Microsoft 365 등

### 🏢 1.1.4 클라우드 구축 모델
* **퍼블릭 클라우드 (Public Cloud) :** AWS, Azure, GCP 등 전문 공급업체가 인터넷을 통해 전 세계 불특정 다수에게 자원을 서비스하는 가장 일반적인 형태입니다.
* **프라이빗 클라우드 (Private Cloud) :** 외부망과 격리되어 특정 기업·기관 단 한 곳만을 위해 독점적으로 구축된 형태입니다. 보안과 규제가 매우 철저한 금융권, 정부 공공기관, 대기업 대형 연구소 등에서 주로 채택합니다.
* **하이브리드 클라우드 (Hybrid Cloud) :** **온프레미스(또는 프라이빗)와 퍼블릭 클라우드를 상호 연동하여 혼합 운영**하는 방식입니다. 실무에서 가장 많이 활용되는 구조입니다. (예: 트래픽 변동이 심한 웹 서버 전면부는 AWS에 두고, 기밀 데이터가 담긴 메인 고성능 DB 서버는 회사 내부 전용 전산실에 유지)

## 🍊 1.2 AWS (Amazon Web Services) 서비스 개요

### 1.2.1 AWS 소개
* **개념 :** 전 세계 클라우드 시장 점유율 1위를 굳건히 지키고 있는 아마존닷컴의 클라우드 플랫폼입니다.
* **주요 고객사 :** 넷플릭스, 트위치, 삼성전자, 쿠팡, 당근마켓 등 국내외 수많은 대형 IT 유니콘 기업들이 인프라의 전반을 AWS 위에서 구동하고 있습니다.
* **핵심 장점 :** 200개가 넘는 압도적으로 다채로운 서비스 스펙트럼을 보유하고 있으며, 완벽한 CLI/API 지원으로 자동화 기능이 탁월하여 **DevOps 친화적인 인프라 환경**을 조성할 수 있습니다.

### 🛠️ 1.2.2 AWS 핵심 서비스 라인업 알아보기
모든 서비스를 다 외울 필요는 없으며, 아래의 10대 핵심 서비스가 인프라 구축의 근간이 됩니다.

1. **EC2 (Elastic Compute Cloud) :** 클릭 몇 번으로 가상 컴퓨터를 대여하는 **가상 서버** 서비스
2. **S3 (Simple Storage Service) :** 용량 제한 없이 이미지, 영상 등 대규모 파일을 안전하게 저장하는 **객체 스토리지**
3. **RDS (Relational Database Service) :** MySQL, PostgreSQL 등을 백업 및 모니터링 자동화 인터페이스로 제공하는 **관계형 관리형 DB**
4. **VPC (Virtual Private Cloud) :** AWS 클라우드 내부에 우리 회사만의 독립된 가상 네트워크 울타리를 구성하는 **네트워크 전용망**
5. **IAM (Identity and Access Management) :** AWS 자원에 접근할 수 있는 사용자 계정을 만들고 권한을 제어하는 **보안 및 인증 관리**
6. **Route 53 :** 가용성이 매우 높은 AWS 전용 표준 **글로벌 DNS 서비스**
7. **ELB (Elastic Load Balancing) :** 다수의 EC2 가상 서버 인프라로 유입 트래픽을 안전하게 분산해 주는 **로드 밸런서**
8. **CloudWatch :** AWS 자원들의 CPU 사용량, 네트워크 트래픽 상태, 에러 로그 등을 실시간 감시하는 **통합 모니터링 툴**
9. **Lambda :** 서버를 상시 켜두지 않고, 특정 이벤트(코드 호출)가 일어날 때만 실행되어 비용을 절끼는 **서버리스(Serverless) 컴퓨팅**
10. **Auto Scaling :** 트래픽이 몰리면 EC2 서버 개수를 자동으로 늘리고(Scale-Out), 한산해지면 줄여주는(Scale-In) **자동 확장 서비스**

### 💳 1.2.3 AWS 과금 체계의 대원칙
* **종량제 과금 (Pay-As-You-Go) :** 사전에 정해진 고정 계약 비용 없이, 수수료처럼 **"실제 사용한 만큼만 초/분/시간 단위로 정산"**하여 지불하는 구조입니다.
* **비용 최적화 팁 :** 테스트용으로 서버를 켠 후 사용하지 않을 때는 **반드시 중지(Stop) 또는 정지(Terminate)** 상태로 만들어야 불필요한 과금 폭탄을 방지할 수 있습니다.

## 🌍 1.3 AWS 글로벌 인프라 구조 (추가 보완)
> 💡 **AWS 가상 인프라가 전 세계 어디에 물리적으로 위치하는지 이해하는 핵심 기본기입니다.**

* **리전 (Region) :** AWS가 전 세계 곳곳에 서비스를 제공하기 위해 격리해 둔 **지리적 허브 구역**입니다. (예: 서울 리전, 도쿄 리전, 버지니아 북부 리전 등)
  * *리전 선택의 기준:* 법적 데이터 규제 준수 여부, 그리고 우리 서비스를 이용할 **최종 사용자와 지리적으로 가장 가까운 곳**을 골라야 네트워크 지연 시간(Latency)을 최소화할 수 있습니다.
* **가용 영역 (AZ, Availability Zone) :** 하나의 리전 내부에 완벽히 독립된 전력, 공조 시스템, 네트워크 인프라를 갖추고 떨어져 있는 **개별 물리 데이터 센터(IDC) 건물 세트**입니다.
  * 하나의 리전은 대개 최소 2~3개 이상의 가용 영역으로 구성됩니다.
  * **실무 활용 :** 시스템 장애에 대응하기 위해 고가용성(HA) 아키텍처를 설계할 때는 웹 서버 두 대를 각각 **서로 다른 가용 영역(예: AZ-A와 AZ-C)에 분산 배치**하여, 하나의 데이터 센터가 낙뢰나 화재로 셧다운되어도 서비스가 중단 없이 유지되도록 설계합니다.

## 🍊 2장 AWS 컴퓨팅 서비스

### 2.1 AWS 컴퓨팅 서비스

#### 2.1.1 컴퓨팅 정의
컴퓨팅이란 데이터를 처리하고 프로그램을 실행하는 능력을 의미합니다. 우리가 서비스하는 쇼핑몰, 게임 서버, 홈페이지 등은 전부 이 컴퓨팅 자원 위에서 실행되기 때문에 클라우드 엔지니어에게 가장 기본이자 중요한 영역입니다.

* **핵심 컴퓨팅 자원:** `CPU`, `Memory(RAM)`, `Storage`, `Network`
* **패러다임의 변화:**
  * **과거 (온프레미스):** 회사 내부에 물리 서버를 구축해야 해서 초기 비용이 많이 들고, 관리가 어려우며, 인프라 확장이 느렸습니다.
  * **현재 (AWS 클라우드):** 컴퓨팅 자원을 필요할 때 필요한 만큼 빌려 씁니다. AWS 환경에서는 EC2를 생성하면 곧바로 서버를 사용할 수 있습니다.

---

#### 2.1.2 AWS 핵심 컴퓨팅 서비스 라인업

AWS는 서비스 목적과 운영 방식에 따라 다양한 형태의 컴퓨팅 환경을 제공합니다.

##### 1. Amazon EC2 (Elastic Compute Cloud)
AWS가 제공하는 가장 대표적인 **가상 서버** 서비스입니다.
* **쉽게 말하면:** 내 PC 대신 AWS 클라우드에 가상 컴퓨터를 생성하고, Ubuntu 같은 OS를 설치하여 웹 서비스를 직접 구동하고 운영할 수 있습니다.
* **실무 활용:** 웹서버, API 서버, DB 서버, Jenkins 서버, Docker 서버 등 대부분의 독립적인 서버 운영에 사용됩니다.

##### 2. AWS Lambda
서버 인프라를 직접 만들거나 관리할 필요 없이 **코드만 업로드하여 실행하는 서버리스(Serverless)** 서비스입니다.
* **예시:** `Python def hello(): return "Hello AWS"`와 같은 코드만 올리면 AWS가 알아서 실행합니다.
* **특징:** 서버 관리 부담이 전혀 없으며, 비용도 고정이 아니라 실제 코드가 실행된 시간과 횟수만큼만 지불하므로 경제적입니다.
* **사용 예:** 자동 알림 발송, 이미지 변환(썸네일 생성), 이벤트 처리 등.

##### 3. Amazon ECS (Elastic Container Service)
AWS에서 **Docker 컨테이너를 실행하고 관리**하는 서비스입니다.
* **역할:** 도커 컨테이너 기반의 애플리케이션을 AWS 환경에서 복잡한 세팅 없이 쉽고 안정적으로 운영하도록 도와줍니다.

##### 4. Amazon EKS (Elastic Kubernetes Service)
컨테이너 오케스트레이션 툴인 **쿠버네티스(Kubernetes)를 AWS에서 편리하게 관리**해 주는 서비스입니다.
* **역할:** 사용자가 직접 쿠버네티스 마스터 노드를 설치하고 관리하는 번거로움 없이, AWS가 검증된 안정적인 쿠버네티스 환경을 대신 운영해 줍니다.

##### 5. AWS Elastic Beanstalk
인프라 구성이 서툰 개발자나 초보자를 위한 **자동 배포 플랫폼**입니다.
* **개념:** 개발자가 소스 코드만 업로드하면, AWS가 알아서 가상 서버(EC2) 생성, 트래픽 분산(로드밸런서), 자동 확장(오토스케일링) 구성을 완전히 자동으로 수행합니다.
* **사용 예:** 간단한 웹 서비스 배포, 테스트용 프로젝트 구동 시 유용합니다.


### 2.2 Amazon EC2 소개
Amazon EC2(Elastic Compute Cloud)는 AWS에서 제공하는 가장 대표적인 가상 서버 서비스(리눅스 또는 윈도우)입니다.

#### 2.2.1 Amazon EC2 인스턴스
* **인스턴스(Instance)란?** AWS 클라우드 상에서 가상으로 실행 중인 하나의 서버 단위를 의미합니다.
* **EC2 인스턴스 상태 종류:**
  * `Running` : 서버가 정상적으로 실행 중인 상태
  * `Stopped` : 서버가 안전하게 정지된 상태 (비용 절감 가능)
  * `Rebooting` : 서버가 재부팅 중인 상태
  * `Terminated` : 서버가 완전히 삭제된 상태 (복구 불가능)

---

#### 2.2.2 AMI (Amazon Machine Image)
* **AMI 란?** 가상 서버를 생성하기 위한 **운영체제(OS) 설치 이미지**입니다.
* **비유:** 컴퓨터를 조립할 때 사용하는 '윈도우 설치 USB'와 같은 역할을 합니다.
* **AMI 종류:** Ubuntu, Amazon Linux, Red Hat Enterprise Linux(RHEL), Windows Server 등
* **서버 생성 흐름:** `AMI 선택` ➔ `EC2 인스턴스 유형/스펙 설정` ➔ `EC2 생성 및 서버 실행`

---

#### 2.2.3 Amazon EC2 스토리지
* **스토리지(Storage)란?** 서버의 데이터를 저장하는 공간으로, 일반 컴퓨터의 하드디스크나 SSD를 의미합니다.
* **EBS (Elastic Block Store):** EC2에서 가장 많이 사용하는 대표적인 가상 드라이브(SSD/HDD) 스토리지입니다.
  * **연결 구조:** `EC2 (컴퓨터 본체)` ➔ (연결) ➔ `EBS (가상 SSD 디스크)`
  * **특징:** EC2 인스턴스를 재부팅하거나 정지하더라도 데이터가 안전하게 유지됩니다.
  * **⚠️ 주의 사항:** 기본 설정의 경우 EC2 인스턴스를 삭제(`Terminated`)할 때 연결된 EBS도 함께 삭제될 수 있으므로, 중요한 데이터는 정기적인 백업(스냅샷)이 필수적입니다.

---

#### 2.2.4 Amazon EC2 네트워킹
EC2 인스턴스는 네트워크 통신을 위해 고유한 IP 주소를 가집니다. 생성 시 목적에 따라 사설 IP와 공인 IP를 함께 부여받을 수 있습니다.

* **사설 IP (Private IP):** AWS 내부 네트워크에서 서버끼리 통신할 때 사용하는 주소입니다. (예: `10.x.x.x`, `172.16.x.x`, `192.168.x.x`)
* **공인 IP (Public IP):** 외부 인터넷 사용자들이 내 서버에 접속할 수 있도록 열려 있는 주소입니다.
* **일반적인 웹 서비스 구조:** `사용자` ➔ `공인 IP 접속` ➔ `EC2 인스턴스 진입` ➔ `Nginx 웹 서버 작동`
* **VPC (Virtual Private Cloud):** 모든 EC2 인스턴스는 격리된 가상 네트워크 공간인 VPC 내부에서 안전하게 실행되어야 합니다.

---

#### 2.2.5 Amazon EC2 보안
* **보안 그룹 (Security Group):** EC2 인스턴스의 앞단을 지키는 **AWS 가상 방화벽**입니다. 허용할 IP 주소와 포트(예: 웹 서비스를 위한 80/443 포트, 원격 접속을 위한 22 포트)를 세밀하게 제어하여 인스턴스를 보호합니다.

---

#### 2.2.6 Amazon EC2 모니터링
* **CloudWatch:** AWS에서 기본으로 제공하는 통합 모니터링 시스템입니다.
* **확인 가능한 핵심 지표:** `CPU 사용률(CPU Utilization)`, `네트워크 수신/송신 트래픽`, `디스크 I/O 사용량`, `인스턴스 상태 검사(Status Check)`

---

#### 2.2.7 Amazon EC2 인스턴스 구입 옵션

비용을 효율적으로 관리할 수 있도록 다양한 서버 구매 방식을 제공합니다.

| 구입 옵션 | 특징 | 장점 | 단점 |
| :--- | :--- | :--- | :--- |
| **1. 온디맨드 (On-Demand)** | 가장 기본적이고 표준적인 옵션 | 초 단위로 사용한 만큼만 깔끔하게 과금 | 약정 할인이 없어 장기 구동 시 비용 부담이 큼 |
| **2. 예약 인스턴스 (RI)** | 1년 또는 3년 기간을 지정하여 이용 | 약정 요금이 적용되어 **비용 할인율이 매우 높음** | 중도 해지가 어려워 장기간 고정적으로 쓸 서버에만 적합 |
| **3. 스팟 인스턴스 (Spot)** | AWS의 남는 유휴 서버 자원을 경매 방식으로 낙찰 | **최대 90% 저렴**하여 압도적인 비용 절감 가능 | AWS에 자원이 부족해지면 **서버가 언제든 강제 회수(종료)**될 수 있음 |

### 2.3 실습: Amazon EC2 인스턴스 배포 및 접근하기
* **실습 목표:** `EC2 생성` ➔ `SSH 접속` ➔ `Nginx 설치` ➔ `웹 서비스 확인` ➔ `모니터링 확인` ➔ `자원 삭제`

#### 2.3.1 AMI를 이용한 EC2 인스턴스 배포하기
* AWS 콘솔에서 원하는 운영체제 이미지(AMI)를 선택한 후, 인스턴스 유형과 키 페어를 설정하여 EC2 가상 서버를 생성합니다.

---

#### 2.3.2 SSH로 EC2 인스턴스에 접속하여 웹 서비스 설정하기
* **SSH(Secure Shell)란?** 네트워크를 통해 원격으로 리눅스 서버에 안전하게 접속하는 기술입니다.

* **인스턴스 접속 및 초기 설정**
  키 파일 권한 변경 (소유자만 읽기 가능하도록 보안 설정)
  > chmod 400 my-key.pem

  SSH를 통해 EC2 인스턴스 원격 접속
  > ssh -i my-key.pem ec2-user@공인IP

  설치된 패키지 최신 버전으로 업데이트
  > sudo dnf update

* **Nginx 웹 서버 설치 및 실행**
  Nginx 웹 서버 패키지 설치
  > sudo dnf install -y nginx

  Nginx 서비스가 정상적으로 실행 중인지 상태 확인
  > sudo systemctl status nginx

  Nginx 웹 서버 서비스 시작
  > sudo systemctl start nginx

---

#### 2.3.3 EC2 인스턴스에 생성된 웹 서비스 접속하기
* 브라우저 주소창에 아래 주소를 입력하여 Nginx 기본 웹 페이지가 잘 뜨는지 확인합니다.
  > http://공인IP

---

#### 2.3.4 EC2 인스턴스 모니터링 설정하기
* AWS의 기본 모니터링 서비스인 **CloudWatch**를 활용합니다.
* **확인 방법:** AWS EC2 콘솔 화면에서 생성한 인스턴스를 선택한 후 하단의 **[모니터링]** 탭을 클릭합니다.
* **CPU 부하 테스트:** CloudWatch의 그래프 변화를 확인하기 위해 의도적으로 CPU 사용량을 100%로 끌어올리는 명령어를 실행합니다.
  > yes > /dev/null

---

#### 2.3.5 리눅스 명령어를 사용하여 EC2 인스턴스 정보 확인하기
인스턴스 내부에서 운영체제 및 하드웨어 자원 상태를 체크하는 유용한 리눅스 명령어 모음입니다.

* **자원 상태 모니터링**
  CPU 및 실행 중인 프로세스 실시간 확인
  > top (또는 htop)

  메모리(RAM) 사용량 확인 (사람이 읽기 쉬운 단위로 출력)
  > free -h

  디스크 남은 용량 및 전체 공간 확인
  > df -h

* **네트워크 및 OS 정보 조회**
  네트워크 인터페이스 및 사설 IP 정보 확인
  > ip addr

  외부에서 바라보는 외부 공인 IP 주소 확인
  > curl ifconfig.me

  현재 설치된 리눅스 운영체제(OS) 상세 정보 확인
  > cat /etc/os-release

  CPU 하드웨어 상세 스펙 확인
  > lscpu

  디스크 장치의 구조 및 볼륨 연결 상태 확인
  > lsblk

---

#### 2.3.6 실습을 위해 생성된 모든 자원 삭제하기
실습이 끝난 후 불필요한 과금을 방지하기 위해 생성한 자원을 순서대로 깔끔하게 정리합니다.

1. **EC2 인스턴스 삭제:** EC2 인스턴스를 선택한 후 **[인스턴스 상태]** ➔ **[인스턴스 종료 (Terminate Instance)]**를 클릭합니다.
2. **EBS 볼륨 확인:** EC2가 삭제되면서 연결되어 있던 EBS 가상 디스크 볼륨도 함께 잘 삭제되었는지 확인합니다.
3. **Elastic IP(고정 공인 IP) 해제:** 만약 실습 중 고정 IP를 할당받아 사용했다면, 인스턴스 종료 후 반드시 **[주소 릴리스]**를 해야 추가 요금이 나오지 않습니다.
4. **스냅샷 확인:** 백업용 스냅샷을 생성했다면 스냅샷 메뉴에서 삭제를 진행합니다.
5. **보안 그룹 정리:** 이번 실습을 위해 임시로 만든 보안 그룹(Security Group)이 있다면 함께 삭제하여 정리합니다.
