# 1장 AWS란?
> 💡 **1장에서는 전통적인 인프라(온프레미스)와 클라우드 컴퓨팅의 차이를 이해하고, AWS의 핵심 서비스 라인업과 인프라 구조를 배웁니다.**

## ☁️ 1.1 클라우드 컴퓨팅 (Cloud Computing)

### 1.1.1 클라우드 컴퓨팅이란?
* **정의 :** 자체적인 물리 서버를 구축하는 대신, 인터넷망을 통해 IT 자원(서버, 스토리지, 네트워크 등)을 **필요한 만큼 빌려 쓰고 비용을 지불하는 서비스**입니다.
* **비유하자면 :** * **온프레미스(On-Premise) :** 대지 매입부터 벽돌 쌓기, 전기/수도 공사까지 직접 수행하여 **집을 짓는 것**
  * **클라우드(Cloud) :** 이미 완벽하게 인프라가 갖춰진 공간에 체크인하여 편하게 머무는 **호텔을 이용하는 것**

### ⚖️ 온프레미스 vs 클라우드 한눈에 비교
| 비교 항목 | 온프레미스 (전통적 방식) | 클라우드 컴퓨팅 (현대적 방식) |
| :--- | :--- | :--- |
| **초기 비용** | 서버 장비 구매, 데이터 센터 구축으로 **막대한 초기 비용** 발생 | 장비 구매가 없으므로 **초기 비용 0원에 수렴** |
| **구축 속도** | 장비 발주, 배송, 서버실 실장까지 **수주~수개월** 소요 | 콘솔 클릭 몇 번으로 **수분 이내**에 인프라 생성 |
| **확장성** | 트래픽 증가 시 추가 서버를 사서 꽂아야 하므로 **증설이 매우 느림** | 클릭 또는 설정에 따라 서버가 실시간 자동 증가 (**Auto Scaling**) |
| **유지 관리** | 전기세, 공조 장치, 고장 난 부품 교체 등 **직접 관리 부담** | 물리 하드웨어 관리 및 인프라 보안은 **클라우드 공급사(AWS)가 전담** |

### 🚀 1.1.2 클라우드 컴퓨팅의 핵심 이점
* **자본 비용의 가변 비용화 :** 대규모 인프라를 미리 구매할 필요가 없어 비즈니스 리스크를 최소화합니다.
* **엄청난 규모의 경제 :** 수많은 가입자가 자원을 공유하므로, 개인이 구축하는 것보다 훨씬 저렴한 단가로 고성능 인프라를 이용할 수 있습니다.
* **용량 예측 불필요 :** 트래픽 수요를 미리 예측해 과도하게 장비를 사둘 필요 없이, 실제 유입량에 맞추어 유연하게 대응합니다.
* **글로벌 배포 가속화 :** 몇 번의 클릭만으로 전 세계(미국, 유럽, 아시아 등)에 서버를 즉시 배포하여 글로벌 서비스를 제공할 수 있습니다.

### 📦 1.1.3 클라우드 컴퓨팅 서비스 유형 (IaaS / PaaS / SaaS)
공급사(AWS)가 어디까지 관리해 주고, 엔지니어가 어디서부터 책임지느냐에 따른 분류입니다.

* **IaaS (Infrastructure as a Service, 인프라 제공 서비스)**
  * **설명 :** 가상화된 하드웨어 자원(서버, 네트워크, 스토리지)만 빌려주는 형태입니다. OS 선택부터 웹 서버, 프로그램 설치 및 환경 설정까지 **엔지니어가 가장 자유롭게 제어**할 수 있습니다.
  * **AWS 대표 서비스 :** **Amazon EC2**, Amazon VPC
  * **실무 예시 :** EC2 서버를 생성한 뒤 Ubuntu OS 위에 Nginx와 Docker를 직접 커맨드로 설치하고 아키텍처를 빌드함.
* **PaaS (Platform as a Service, 플랫폼 제공 서비스)**
  * **설명 :** 밑바탕이 되는 인프라와 OS, 런타임 환경까지 인프라 가상화 플랫폼이 전부 알아서 관리해 줍니다. 개발자는 하드웨어 신경 세팅 없이 **오직 '애플리케이션 코드'만 업로드**하면 됩니다.
  * **AWS 대표 서비스 :** **AWS Elastic Beanstalk**, AWS Lambda
  * **실무 예시 :** 개발한 백엔드 소스 코드를 압축해서 업로드하면, AWS가 알아서 서버 대수 조절과 로드 밸런싱을 수행함.
* **SaaS (Software as a Service, 소프트웨어 제공 서비스)**
  * **설명 :** 클라우드 공급업체가 인프라부터 최종 소프트웨어 애플리케이션까지 **100% 완성하여 서비스 형태로 제공**하는 모델입니다. 사용자는 별도의 설치 없이 계정 로그인만 해서 기능을 이용합니다.
  * **대표 예시 :** Gmail, Slack, Notion, Microsoft 365 등

### 🏢 1.1.4 클라우드 구축 모델
* **퍼블릭 클라우드 (Public Cloud) :** AWS, Azure, GCP 등 전문 공급업체가 인터넷을 통해 전 세계 불특정 다수에게 자원을 서비스하는 가장 일반적인 형태입니다.
* **프라이빗 클라우드 (Private Cloud) :** 외부망과 격리되어 특정 기업·기관 단 한 곳만을 위해 독점적으로 구축된 형태입니다. 보안과 규제가 매우 철저한 금융권, 정부 공공기관, 대기업 대형 연구소 등에서 주로 채택합니다.
* **하이브리드 클라우드 (Hybrid Cloud) :** **온프레미스(또는 프라이빗)와 퍼블릭 클라우드를 상호 연동하여 혼합 운영**하는 방식입니다. 실무에서 가장 많이 활용되는 구조입니다. (예: 트래픽 변동이 심한 웹 서버 전면부는 AWS에 두고, 기밀 데이터가 담긴 메인 고성능 DB 서버는 회사 내부 전용 전산실에 유지)

## 🍊 1.2 AWS (Amazon Web Services) 서비스 개요

### 1.2.1 AWS 소개
* **개념 :** 전 세계 클라우드 시장 점유율 1위를 굳건히 지키고 있는 아마존닷컴의 클라우드 플랫폼입니다.
* **주요 고객사 :** 넷플릭스, 트위치, 삼성전자, 쿠팡, 당근마켓 등 국내외 수많은 대형 IT 유니콘 기업들이 인프라의 전반을 AWS 위에서 구동하고 있습니다.
* **핵심 장점 :** 200개가 넘는 압도적으로 다채로운 서비스 스펙트럼을 보유하고 있으며, 완벽한 CLI/API 지원으로 자동화 기능이 탁월하여 **DevOps 친화적인 인프라 환경**을 조성할 수 있습니다.

### 🛠️ 1.2.2 AWS 핵심 서비스 라인업 알아보기
모든 서비스를 다 외울 필요는 없으며, 아래의 10대 핵심 서비스가 인프라 구축의 근간이 됩니다.

1. **EC2 (Elastic Compute Cloud) :** 클릭 몇 번으로 가상 컴퓨터를 대여하는 **가상 서버** 서비스
2. **S3 (Simple Storage Service) :** 용량 제한 없이 이미지, 영상 등 대규모 파일을 안전하게 저장하는 **객체 스토리지**
3. **RDS (Relational Database Service) :** MySQL, PostgreSQL 등을 백업 및 모니터링 자동화 인터페이스로 제공하는 **관계형 관리형 DB**
4. **VPC (Virtual Private Cloud) :** AWS 클라우드 내부에 우리 회사만의 독립된 가상 네트워크 울타리를 구성하는 **네트워크 전용망**
5. **IAM (Identity and Access Management) :** AWS 자원에 접근할 수 있는 사용자 계정을 만들고 권한을 제어하는 **보안 및 인증 관리**
6. **Route 53 :** 가용성이 매우 높은 AWS 전용 표준 **글로벌 DNS 서비스**
7. **ELB (Elastic Load Balancing) :** 다수의 EC2 가상 서버 인프라로 유입 트래픽을 안전하게 분산해 주는 **로드 밸런서**
8. **CloudWatch :** AWS 자원들의 CPU 사용량, 네트워크 트래픽 상태, 에러 로그 등을 실시간 감시하는 **통합 모니터링 툴**
9. **Lambda :** 서버를 상시 켜두지 않고, 특정 이벤트(코드 호출)가 일어날 때만 실행되어 비용을 절끼는 **서버리스(Serverless) 컴퓨팅**
10. **Auto Scaling :** 트래픽이 몰리면 EC2 서버 개수를 자동으로 늘리고(Scale-Out), 한산해지면 줄여주는(Scale-In) **자동 확장 서비스**

### 💳 1.2.3 AWS 과금 체계의 대원칙
* **종량제 과금 (Pay-As-You-Go) :** 사전에 정해진 고정 계약 비용 없이, 수수료처럼 **"실제 사용한 만큼만 초/분/시간 단위로 정산"**하여 지불하는 구조입니다.
* **비용 최적화 팁 :** 테스트용으로 서버를 켠 후 사용하지 않을 때는 **반드시 중지(Stop) 또는 정지(Terminate)** 상태로 만들어야 불필요한 과금 폭탄을 방지할 수 있습니다.

## 🌍 1.3 AWS 글로벌 인프라 구조 (추가 보완)
> 💡 **AWS 가상 인프라가 전 세계 어디에 물리적으로 위치하는지 이해하는 핵심 기본기입니다.**

* **리전 (Region) :** AWS가 전 세계 곳곳에 서비스를 제공하기 위해 격리해 둔 **지리적 허브 구역**입니다. (예: 서울 리전, 도쿄 리전, 버지니아 북부 리전 등)
  * *리전 선택의 기준:* 법적 데이터 규제 준수 여부, 그리고 우리 서비스를 이용할 **최종 사용자와 지리적으로 가장 가까운 곳**을 골라야 네트워크 지연 시간(Latency)을 최소화할 수 있습니다.
* **가용 영역 (AZ, Availability Zone) :** 하나의 리전 내부에 완벽히 독립된 전력, 공조 시스템, 네트워크 인프라를 갖추고 떨어져 있는 **개별 물리 데이터 센터(IDC) 건물 세트**입니다.
  * 하나의 리전은 대개 최소 2~3개 이상의 가용 영역으로 구성됩니다.
  * **실무 활용 :** 시스템 장애에 대응하기 위해 고가용성(HA) 아키텍처를 설계할 때는 웹 서버 두 대를 각각 **서로 다른 가용 영역(예: AZ-A와 AZ-C)에 분산 배치**하여, 하나의 데이터 센터가 낙뢰나 화재로 셧다운되어도 서비스가 중단 없이 유지되도록 설계합니다.
